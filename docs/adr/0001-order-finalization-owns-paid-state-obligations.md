# Order Finalization Owns Paid-State Obligations

Power Plunge will treat Order finalization as the business transition where a payable Order becomes paid and post-payment obligations become owed, not as behavior owned by any one Stripe route. All trusted triggers, including Stripe webhooks and server-verified customer completion, must route through one Order finalization service. Stripe event dedupe only protects external event delivery; business idempotency comes from a durable pending-to-paid finalization claim on the Order. For the first implementation slice, we will avoid adding a full obligation ledger schema and instead use a no-schema service that conditionally claims `pending -> paid`, then performs the current obligations. This is a trade-off: lower migration and delivery cost now, with less resumability until a later ledger-backed implementation.
